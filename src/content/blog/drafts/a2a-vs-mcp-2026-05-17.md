跟同一個 AI 講話，有時候感覺它什麼都做得到，有時候又明顯撞牆。

大多數人都有類似的體驗。Claude 或 ChatGPT 在聊天視窗裡回答問題可以很厲害，但換一個場景它就突然變笨了。或者，你用 Claude 整理了一份分析，想交給另一個 AI 工具繼續做下一步，卻發現這兩個 AI 根本沒有辦法直接「對話」，你只能複製貼上，親自當翻譯。

這兩個問題，背後是兩件不同的事。

有人在解第一件事——讓 AI 能真正「動手做事」，不只是說話。這個解法叫 **MCP**。有人在解第二件事——讓多個 AI 之間能協作分工、互相交接任務。這個解法叫 **A2A**。

一個是 AI 的**左手**，一個是 AI 的**無線電**。

---

## 先把比喻講透

AI 模型本來只能輸出文字。給它一個問題，它給你一個答案。這就是一個沒有左手的存在——可以思考、可以說話，但沒辦法伸手拿東西、沒辦法按按鈕、沒辦法翻看文件。

**MCP 是那隻左手**。

Model Context Protocol 是 Anthropic 在 2024 年底[開源的協議](https://www.anthropic.com/news/model-context-protocol)。它讓 AI 可以呼叫外部工具——查資料庫、搜尋網路、操作瀏覽器、讀取程式碼庫。這隻手伸出去，抓住了外部世界。方向很明確：單向，從 AI 往外。

左手做好了，一個 AI 可以做很多事了。但下一個問題來了：如果有兩個、五個、十個 AI，它們要怎麼協作？誰負責哪一段？任務怎麼交接？

這時候需要的不是手，是**無線電**。

**A2A 是那個無線電頻道**。

Agent2Agent 是 Google 在 2025 年[發表的協議](https://developers.googleblog.com/en/a2a-a-new-era-of-agent-interoperability/)。它定義的是多個 AI agent 之間怎麼找到彼此、傳遞訊息、委派任務。一個 agent 負責查資料，一個 agent 負責生成，一個 agent 負責審核——靠 A2A 這個共同頻道，各自完成自己的部分，結果再匯攏。

左手讓 AI 能拿工具。無線電讓 AI 能組隊。兩件不同的事，兩個不同的協議。

---

## MCP — 我每天都在用的那個

我的 Claude Code 現在同時連著三個 MCP server：Serena（語意化程式碼分析工具）、Pencil（設計工具）、Chrome（瀏覽器自動化）。這三個工具各自來自不同公司，但 Claude Code 接起來的方式完全一樣，因為它們都遵循 MCP 的規格。

這就是 MCP 在解的問題——不是讓 AI 更聰明，是讓它的左手能接上不同廠牌的工具。

你可以把 MCP 想成一個插座標準。以前每個工具做自己的介面，AI 要接每個工具都要學一次新語言。MCP 出來之後，工具廠做一個 MCP server，AI 用同一套方式就能操作所有支援 MCP 的工具。

MCP 在 2025 年的生態已經非常成熟。主流開發工具、資料庫、設計工具、瀏覽器自動化，幾乎都有 MCP server 可以接。如果只是讓一個 AI agent 能動手做事，MCP 就夠了。

但有一件事 MCP 沒辦法解：一個 agent 怎麼跟另一個 agent 說話。

---

## A2A — 為什麼現在大家突然都在談

2025 年，Google 丟出了 A2A，同時帶了超過 50 個技術夥伴一起亮相，名單裡有 Salesforce、SAP、Atlassian，還有 Accenture 這類大型顧問公司。幾個月後，Google [把 A2A 捐給了 Linux Foundation](https://developers.googleblog.com/en/google-cloud-donates-a2a-to-linux-foundation/)——從 Google 內部的提案，變成行業標準的候選人。

這種規格捐給 Linux Foundation 的動作，意思很清楚：他們不想要它只是 Google 的東西，他們想要它成為整個行業的共同語言。

為什麼需要這個共同語言？

因為多個 AI 協作，真正的困難不是技術，是**語言不通**。

想像一個顧問公司裡，有分析師、有寫手、有審稿、有客戶窗口。他們要協作，需要共同的工作語言——任務怎麼描述、交接給誰、完成的標準是什麼。沒有共同語言，就算每個人都很強，團隊還是會開始互相等待、誤解、重工。

AI agent 的問題一模一樣。每個 agent 都有自己的能力，但如果沒有共同的協議，它們就沒辦法找到彼此、委派任務、確認完成。A2A 做的事，就是給這個協作提供一套共同語言。

LangGraph、CrewAI、LlamaIndex 這些主流的 multi-agent framework，目前都支援 A2A。生態在成形，但真正生產部署的多 agent 系統，很多還是在用自己的 orchestration 邏輯，A2A 普及需要時間。

但「自己 orchestration 跑得通」這件事，遲早會撞到一個邊界——當你的 agent 需要跟**不屬於你這套系統**的另一個 agent 對話的時候。

---

## 插播：別把 A2A 旁邊三個東西當成同一個

A2A 周圍有三個東西很容易被誤認成同一件事，先拆清楚。

**IBM 的 ACP（Agent Communication Protocol）**：2025 上半年 IBM 推的競品，跟 A2A 想解同一件事（agent 跨主體互通）。下半年[已經併入 A2A 治理](https://lfaidata.foundation/communityblog/2025/08/29/acp-joins-forces-with-a2a-under-the-linux-foundations-lf-ai-data/)，獨立路線停止發展。今天看到「IBM ACP」基本上等於 A2A 的歷史前身。

**Anthropic 的 ACP（Agent Client Protocol）**：純粹撞名意外。它管的是 Zed 編輯器這種 IDE 怎麼跟裡面跑的 coding agent 對話，是**單一系統內**的 IDE ↔ agent 介面，跟「agent 跨主體互通」完全不同層次。讀到「ACP」時先確認是誰的 ACP。

**「兩個 LLM 用 prompt 互相對話」也常被叫 A2A**：從**概念**上你說它算 agent-to-agent 沒問題；但從 **A2A protocol** 的嚴格定義，要符合 task statement、capability discovery、UX negotiation、security policy 才算。你用 Claude Code 派 subagent 互相支援、用 OpenAI Swarm 做 handoff、用 LangGraph 寫 multi-agent 流程——這些都是 multi-agent orchestration（同一系統內的編排），不是 A2A protocol（跨主體網路通訊）。生態剛開始，很多自稱用了 A2A 的系統其實是 proprietary workflow，讀到時先確認是哪一個層級。

## 我自己的系統，以及為什麼還沒接 A2A

我現在跑的 creative-team 有 6 個 agent：有人負責研究、有人負責起草、有人負責審稿、有人負責 fact-check。這篇文章，就是在這個系統裡生成的。

但這個系統目前沒有用 A2A。

agent 之間的協調，全部靠主 Claude 路由——主 Claude 決定派誰去做什麼，自己充當中介層。這是現實：有標準不代表所有人都立刻跑去採用標準，特別是當自己的方案已經跑得通的時候。

這套自建的路由跑了幾個月，大部分情況都還順。但偶爾我會撞到一個明顯的天花板：**子 agent 不能再派子 agent**。

寫稿這個 agent 如果在執行中途發現缺了一段資料，沒辦法自己叫研究員來補。它得先停下來、把結果交還給主 Claude、主 Claude 看到缺口、再派研究員、研究員做完、主 Claude 再把資料丟回給寫稿 agent。多繞了三步，每步都要主 Claude 介入一次。

這就是一個 A2A 想解決的問題——讓 agent 能主動找其他 agent 喊話，而不是全部靠中央上層統一排程。

不過這也說明了一件事：A2A 的真正價值不在「兩個 agent 互相喊話」這麼簡單，而在「跨主體、跨平台」的場景。當你的 agent 需要呼叫**另一間公司**的 agent，沒有共同協議的時候，你只能各自搭橋。A2A 提供的是那座橋的標準規格。

## 為什麼 A2A 終究會變必要

想像你有一個 personal health agent，跟著你一輩子。它知道你的病史、用藥、過敏、家族遺傳、健身習慣、最近的睡眠數據。它不屬於任何一間醫院、任何一間保險公司、任何一個 app——它是你的。

今天你去 A 診所看感冒，醫師需要知道你最近的用藥；明天你到 B 醫院做檢查，放射科要拉你的影像史；後天保險公司審你的理賠單，需要 cross-check 過往記錄。三個機構，三個系統，三個各自的 agent。

**沒有共同協議的世界**：你拍病歷給 B 醫院、複製貼上保險公司表格、自己當人肉中介。這就是現在的醫療資訊流。

**有 A2A 的世界**：你的 personal agent 跟 A 診所、B 醫院、保險公司的 agent 用同一套協議對話。誰需要什麼、誰提供什麼、什麼資料能傳、什麼不能、怎麼確認身份、怎麼留稽核軌跡——這些都有共同語言。你還是有否決權，但不用再當資訊搬運工。

這個場景不是科幻。Personal agent 的雛形已經在做了。當它成熟、當機構也開始有自己的 agent——它們必然要對話，必然會跨組織、跨 vendor、跨信任域。

這就是 A2A 的真正戰場。不是「兩個 agent 互相喊話」這麼小，而是**個人 agent 跟機構 agent 之間的標準介面**。今天感覺很遠，是因為 personal agent 還沒普及。但同樣的故事在網路時代發生過——1995 年 email 跨服務商互通看起來像奢侈，三年後變成預設。

## 什麼時候用哪個，還有第三選項

整理一下判斷框架：

**用 MCP**：你的 agent 需要呼叫外部工具。讀資料庫、查 API、操作瀏覽器——只要是「agent 伸手拿東西」，MCP 是對的工具。現在生態成熟，主流工具都有 MCP server。

**用 A2A**：系統裡有多個 agent，需要彼此發現、溝通、委派任務。特別是跨公司、跨平台的協作場景——你的 agent 需要呼叫另一間公司的 agent 時，有標準語言比各自搭橋好太多。

但還有**第三個選項**：CLI。

我的 Calendar 操作用 `gws` CLI。Email 操作用 `email_ops.py`。這些不是 MCP，也不是 A2A。是已經存在、已經測試過、有結構化輸出的命令列工具，agent 直接呼叫，效果一樣。

工程師社群裡有一個常見的觀察：對於簡單的查詢和操作，CLI + JSON output 比搭一個 MCP server 更輕、更快。而且人和 agent 用同一套工具——除錯的時候不需要切換思維，方便很多。

選工具前先問「這個操作有多複雜」：

- 簡單的查詢和 CRUD → CLI 就夠，不需要 MCP
- 需要雙向通訊、streaming、上下文管理 → MCP
- 多個 agent 需要協同、跨系統任務分派 → A2A

三個選項，不是二選一的題目。

## 這跟你的工作有什麼關係

如果不寫程式、不建 AI 系統，MCP 和 A2A 聽起來很遠。

但它們解的問題，其實很熟悉。

一個有能力的人，如果沒有工具可以用，能力只能停在腦子裡。MCP 解的就是這個問題——讓能力能真正「接觸」到外部世界。工具鏈好不好，決定了一個人（或一個 AI）有多少地方能真正動手。

多個有能力的人，如果沒有共同語言，再強的個人加在一起也是一盤散沙。A2A 解的就是這個問題——跨主體的協作，需要一套共同的交接語言。誰負責什麼、怎麼交代清楚、完成的標準是什麼。這是每一間公司、每一個跨部門專案、每一個分工協作都需要解決的事。

我在顧問業務裡常看到這兩個問題同時出現。個人能力很強，但工具鏈斷掉——能說但不能做，想法停在會議記錄裡。跨部門協作有共識，但沒有交接語言——每次任務切換都要重新說一遍，前一段做的東西後一段的人不知道。

AI agent 的問題跟人的問題，結構是一樣的。工具鏈讓個體能動手，協作語言讓群體能分工。兩件事都要有，少一件都跑不起來。

MCP 和 A2A 現在在解的，是 AI 版本的這個問題。它們還在演化，生態還在成形。但它們想解的問題，已經存在很久了。
